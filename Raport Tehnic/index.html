Introducere

LoFo38 este o aplicație disponibilă Web care permite utilizatorilor înregistraţi să îşi caute un obiect pierdut în speranţă recuperării lui sau să raporteze găsirea unui obiect necunoscut cu scopul de a-i fi înmânat înapoi posesorul. Şi totodată oferă acestora o modalitate de comunicare între ei, astfel obiectul pierdut poate ajunge înapoi în posesia deţinătorului de drept.
Scopul aplicaţiei: localizarea posesorului de drept al unui obiect pierdut, utilizând metoda de comunitare prin mesaje, sau telefonul mobil.

Detalii despre aplicație

  • Reprezentare utilizator: Un cont de utilizator, creat în cadrul site-ului, prin metoda de înregistrare user/pass.
  • Reprezentare obiectelor pierdute: Pagina ce curpinde funcţionalitatea principală a aplicaţiei în care utilizatorul va putea alege categoria din care face parte obiectul pierdut (printr-o listă dropdown) şi alege semnele distincte prin input-uri, adăugarea unei poze după care poate fi localizat obiectul respectiv.
  • Reprezentare obiectelor gasite: Pagina ce curpinde funcţionalitatea principală a aplicaţiei în care utilizatorul va putea alege categoria din care face parte obiectul pierdut (printr-o listă dropdown) şi alege semnele distincte prin input-uri, adăugarea unei poze după care poate fi localizat obiectul respectiv.
  • Reprezentare sistemului de mesaje: Pagina noastră cuprinde şi un sistem de mesaje între utilizatori, de tipul "Send/Reply" unde un utilizator care a pierdut un obiect îl poate contacta pe cel care l-a pierdut în vederea recuperării obiectului.

AJAX is a set of Web development techniques using many Web technologies on the client side to create asynchronous Web applications. With Ajax, Web applications can send data to and retrieve from a server asynchronously (in the background) without interfering with the display and behavior of the existing page. By decoupling the data interchange layer from the presentation layer, Ajax allows for Web pages, and by extension Web applications, to change content dynamically without the need to reload the entire page. In practice, modern implementations commonly substitute JSON for XML due to the advantages of being native to JavaScript.

Server-ul de baze de date utilizat (mySQL)

Pentru a reține toți utilizatorii înregistraţi și datele introduse de aceştia în aplicaţia web vom avea nevoie de o bază de date MySQL:

  • Tabela found_objects: va conține următoarele atribute: id(PK, A_I), username, category, description, picture_location, location, data, is_verified, status asociate fiecărui obiect care a fost raportat drept găsit de un username.
  • Tabela found_ob_tags: va conține următoarele atribute: id(PK, A_I), obj_id, tag asociate descrierii fiecărui obiect care a fost raportat drept pierdut de un username.
  • Tabela lost_objects: va conține următoarele atribute: id(PK, A_I), username, category, description, picture_location, location, data, is_verified, status asociate fiecărui obiect care a fost raportat drept gasit de un username.
  • Tabela lost_ob_tags: va conține următoarele atribute: id(PK, A_I), obj_id, tag asociate descrierii fiecărui obiect care a fost raportat drept pierdut de un username.
  • Tabela pm: va conține atributele: msg_id(PK, A_I), id, id2, title, user1, user2, message, timestamp, user1read, user2read ascociate fiecărui mesaj trimis de useri prin intermediul sistemului de mesaje.
  • Tabela reported_by: va conține atributele: report_id(PK, A_I), reported_username, reason, described_report, reported_by, report_time, solved asociate fiecărui report trimis de câtre un user.
  • Tabela users: va conține atributele: user_id(PK, A_I), user_token, username, last_name, first_name, email, , pass, , address, phone_number, role care va face legătura între toate cele 4 tabele menţionate mai sus.

Limbaje de programare utilizate

  1. Pentru interfața grafică: se va utiliza HTML, CSS şi JS.
  2. Pentru interogarea bazei de date se va utiliza SQL.
  3. Pentru exportarea rapoartelor cu privire la returnarea obiectelor proprietarilor se va folosi JSON, PDF, CSV şi HTML.

Arhitectura aplicației

  • Aplicaţia noastră "simulează" un departament de lucruri pierdute, lucrurile fiind introduse într-o bază de date de tip mySQL care va fi interogată de către membri ce urmează a-şi căuta obiectele pierdute.
    • Aplicația noastră are următoarele pagini publice:
  • pagina de pornire - unde vizitatorul alege dacă se poate loga, înregistra apoi poate alege dacă a pierdut/ găsit un obiect.
  • pagina de înregistrare, în care utilizatorul introduce datele cu care doreşte să-şi facă cont.
  • pagina de profil personal, în care utilizatorul îşi poate modificare informaţiile profilului.
  • pagina de căutare/adăugare obiecte pierdute - unde utilizatorul specifică detalii legate de obiectul pe care l-a pierdut.
  • pagina de adăugare obiect pierdute - unde utilizatorul ce a găsit un obiect străin poate specifica detaliile acestuia.
  • pagina de afişare a obiectului - unde utilizatorul poate afla informaţii despre obiectul care pare a fi similar informaţiilor introduse în pagina de căutare obiect.
  • pagina de report - unde orice utilizator poate raporta un abuz referitor la un obiect ce este postat/pierdut/găsit şi faţă de care poate face dovadă proprităţii.
  • pagina de contact us - unde orice utilizator poate intra în contact cu noi, printr-un form de contactare.
  • pagina de mesaje personale - unde orice utilizator poate vedea mesajele lui.
  • pagina de obiecte personale - unde orice utilizator poate vedea obiectele pierudte/găsite de el.
    • De asemenea următoarele pagini sunt disponibile pentru admin:
  • pagina de aprobat obiecte pierdute - unde admin-ul poate aproba obiectele pierdute înainte ca acestea să apară pe pagină.
  • pagina de aprobat obiecte găsite - unde admin-ul poate aproba obiectele găsite înainte ca acestea să apară pe pagină.
  • pagina de utilizatori - unde adimn-ul poate vedea informaţii despre alţi utilizatori şi le poate modifica rank-ul.
  • pagina de consultat report-uri - unde admin-ul poate vedea ce report-uri s-au dat pe site.
  • Modul de funcţionare: vizitatorul acceasză pagina de pornire a aplicaţiei unde se poate loga sau înregistra. Dacă informaţiile introduse în câmpurile de login corespund cu informaţiile prezente în baza de date atunci login-ul are succes. Dacă nu atunci acesta este atenţionat şi i se pune la dispoziţie crearea de cont sau recuperarea parolei contului. Dacă utilizatorul alege să se înregistreze atunci este nevoit să introducă informaţiile pe care doreşte să le aibă contul său. După verificarea cu succes a informaţiilor introduse, contul este creat şi este permisă logarea. După logare, utilizatorul poate specifica informaţii legate despre obiectul pierdut accesând ramura de "obiecte pierdute" a aplicaţiei web, iar aceste informaţii vor fi comparate cu cele prezente în baza de date, iar dacă există similarităţi cu privire la informaţiile introduse şi cele prezente în baza de date obiectul este prezentat utilizatorului.

    • Arhitectura de ansamblu

    Interfața Grafică:
    • Pe pagina de pornire a aplicaţiei web nu avem un loc specifici pentru login acesta fiind implementat în bannerul prezentat în partea de header a aplicaţiei.

    • Dacă vizitatorul încearcă să acceseze pagini care necesită un cont, va primi un mesaj de eroare ce-i va atrage atenţia asupra faptului ca trebuie să fie logat.

    • După logare, meniul este "deschis" total funcţiilor de membri şi nu numai. Utilizatorul poate folosi funcţia de mesaje, de report şi îşi poate vedea obiectele adăugate.
